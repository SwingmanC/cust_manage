## 1. 文档概述

### 1.1 建设背景

苏州移动战略客户中心需要建设“一企一册”集团客户信息化管理系统，对战略客户、冠名集团、收入经营、业务渗透、通讯录、友商竞对和商机信息进行统一沉淀和管理。当前需求文档已经提出客户经理、BU 长、管理员三类用户视角，并要求系统基于 SSM 后端、Vue 前端建设，测试阶段使用 MySQL，后续切换到公司 OceanBase 数据库。

同时，原需求第 4 节“未来构想”提出：公司各条线存在类似信息化管理需求，后续计划搭建公司级后台管理系统，统筹各条线的数据和流程管理，公司员工从统一入口进入各自所属部门的平台。因此，本设计既要满足战略客户中心“一企一册”的当前业务需求，也要为未来统一门户、多条线接入和流程中心预留扩展能力。

### 1.2 建设目标

1. 建立集团客户“一企一册”数据档案，覆盖集团基本信息、收入、BC 业务、友商竞对、商机、通讯录等维度。
2. 为客户经理提供本人所辖集团的数据看板、详情查询、通讯录录入和竞对信息录入能力。
3. 为 BU 长提供 BU 级经营看板、下属集团查询、通讯录审核和竞对信息维护能力。
4. 为管理员提供全局数据看板、全量集团管理、数据导出、基础配置、权限管理和日志审计能力。
5. 建立规范数据库模型，支持 MySQL 与 OceanBase 灵活切换。
6. 建立统一门户扩展模型，支持未来公司各条线平台从统一入口接入。

### 1.3 系统范围

本系统当前范围包括：

- 用户登录与权限控制。
- 客户经理端功能。
- BU 长端功能。
- 管理员端功能。
- 集团客户主数据管理。
- 收入数据同步与看板展示。
- 通讯录录入、导入、审核和生效。
- 友商竞对信息录入、维护和留痕。
- 商机信息查询与展示。
- 基础数据、字典、角色、菜单、数据权限配置。
- 登录日志、操作日志、数据导出。

未来扩展范围包括：

- 公司级统一门户。
- 多条线/多部门平台接入。
- 跨平台应用授权。
- 统一流程中心。
- 统一报表与数据看板中心。

### 1.4 用户角色

| 角色 | 角色说明 | 数据范围 |
|---|---|---|
| 客户经理 | 负责维护本人名下集团客户信息 | 本人负责的有效冠名集团 |
| BU 长 | 管理所属 BU 下客户经理和集团客户 | 本 BU 及下属客户经理负责的集团 |
| 管理员 | 系统管理和全量数据管理人员 | 全系统数据 |
| 平台管理员 | 未来统一门户下的平台配置人员 | 指定应用或条线范围 |
| 审计人员 | 查看日志、导出记录和操作轨迹 | 按授权查看 |

### 1.5 关键术语

| 术语 | 说明 |
|---|---|
| 一企一册 | 以单个集团客户为单位，汇聚该集团基础资料、经营收入、业务情况、通讯录、竞对、商机等数据的管理模式 |
| 集团客户 | 政企客户或战略客户中心管理的集团单位 |
| 冠名集团 | 已明确归属客户经理或 BU 管理的集团客户 |
| BU | 战略客户中心内部业务单元或行业条线，如交通能源、公检法军等 |
| BC 业务 | 面向集团客户成员或业务场景统计的业务指标 |
| 友商竞对 | 电信、联通、广电等竞争方在集团客户中的业务情况 |
| 统一门户 | 公司级统一入口，员工从该入口进入其有权限访问的部门平台 |

## 2. 总体功能架构

### 2.1 功能模块总览

系统功能分为当前一企一册核心能力和未来公司级后台扩展能力两大部分。

```text
一企一册系统
├── 统一入口与登录
│   ├── 账号/OA 登录
│   ├── 权限加载
│   ├── 菜单路由生成
│   └── 应用入口展示
├── 客户经理端
│   ├── 个人数据看板
│   ├── 分集团查询
│   ├── 集团详情
│   ├── 通讯录管理
│   ├── 友商竞对录入
│   └── 商机查看
├── BU 长端
│   ├── BU 数据看板
│   ├── BU 集团查询
│   ├── 通讯录上传
│   ├── 通讯录审核
│   └── 友商竞对管理
├── 管理员端
│   ├── 全局数据看板
│   ├── 全量集团管理
│   ├── 数据导出
│   ├── 用户与组织管理
│   ├── 角色与权限管理
│   ├── 字典配置
│   └── 日志审计
├── 数据与流程能力
│   ├── 收入数据同步
│   ├── Excel 导入导出
│   ├── 通用工单
│   └── 审批记录
└── 未来统一后台能力
    ├── 应用/平台管理
    ├── 条线管理
    ├── 应用授权
    ├── 统一流程中心
    └── 统一报表中心
```

### 2.2 业务对象关系

核心业务对象关系如下：

1. 一个 BU/部门下有多个客户经理。
2. 一个客户经理负责多个集团客户。
3. 一个集团客户有多条通讯录信息、月度收入、BC 业务记录、友商竞对记录和商机信息。
4. 一个集团客户可与其他集团形成母子、附属、统付等关系。
5. 通讯录和友商信息的新增/修改可触发工单审核。
6. 用户通过角色获得菜单和操作权限，通过数据权限控制可见集团范围。
7. 未来每个业务平台作为一个应用接入统一门户，应用拥有独立菜单、角色和数据域。

## 3. 功能点详细说明

## 3.1 统一入口与登录

### 3.1.1 用户登录

| 项目 | 内容 |
|---|---|
| 功能说明 | 用户通过 OA、统一认证或本地账号登录系统 |
| 使用角色 | 客户经理、BU 长、管理员、平台管理员 |
| 前置条件 | 用户账号已在系统中启用，并已绑定组织、角色和数据权限 |
| 输入 | OA 账号/用户名、密码或统一认证票据 |
| 处理规则 | 校验账号状态；加载用户角色、菜单权限、数据范围；记录登录日志 |
| 输出 | 登录成功后进入首页；登录失败提示原因 |
| 权限控制 | 未登录用户不得访问业务接口；禁用账号不得登录 |
| 异常处理 | 账号不存在、密码错误、账号停用、权限为空时分别提示，并记录登录失败日志 |

### 3.1.2 权限菜单加载

| 项目 | 内容 |
|---|---|
| 功能说明 | 根据用户角色和应用权限生成前端菜单、按钮权限和接口权限 |
| 使用角色 | 所有登录用户 |
| 前置条件 | 用户已登录 |
| 输入 | 用户 ID、当前应用 ID |
| 处理规则 | 查询用户角色；合并角色菜单；过滤停用菜单；生成前端路由和权限标识 |
| 输出 | 菜单树、按钮权限集合、数据范围 |
| 权限控制 | 用户只能看到授权应用和授权菜单 |
| 异常处理 | 无任何菜单权限时进入无权限提示页 |

### 3.1.3 统一应用入口

| 项目 | 内容 |
|---|---|
| 功能说明 | 展示用户可访问的平台入口，例如一企一册系统、未来其他条线系统 |
| 使用角色 | 所有登录用户 |
| 前置条件 | 统一门户启用，用户已被授权至少一个应用 |
| 输入 | 用户 ID、组织 ID |
| 处理规则 | 查询用户直接授权和部门授权的应用；按应用状态和排序展示 |
| 输出 | 应用卡片、应用名称、应用说明、入口地址 |
| 权限控制 | 未授权应用不可见、不可访问 |
| 异常处理 | 应用停用时提示“应用暂不可用” |

## 3.2 客户经理端功能

### 3.2.1 客户经理数据看板

| 项目 | 内容 |
|---|---|
| 功能说明 | 展示当前客户经理负责集团的核心经营指标 |
| 使用角色 | 客户经理 |
| 数据范围 | 当前登录客户经理本人负责的有效集团 |
| 核心指标 | 冠名集团数、上月收入、2026 年累计收入趋势、重点业务收入拆分、通讯录完善情况、竞对信息覆盖情况 |
| 处理规则 | 按客户经理 ID 过滤集团；按收入月份汇总收入；按业务科目拆分收入；折线图展示年度累计趋势 |
| 输出 | 指标卡片、收入折线图、业务收入柱状图、待处理提醒 |
| 异常处理 | 无集团时展示空状态；收入未同步时展示“暂无收入数据” |

### 3.2.2 分集团情况查询

| 项目 | 内容 |
|---|---|
| 功能说明 | 客户经理查询本人名下所有集团，并进入单集团详情页 |
| 使用角色 | 客户经理 |
| 查询条件 | 集团名称、集团编号、集团类型、是否重要集团、是否要客、是否 139、本地/离岸 |
| 处理规则 | 查询结果强制按当前客户经理数据范围过滤；支持精准查询和模糊查询 |
| 输出 | 集团列表、分页结果、集团标签、客户经理、BU、联系人、收入概览 |
| 权限控制 | 客户经理不得查看非本人集团详情 |
| 异常处理 | 查询无结果时展示空列表 |

### 3.2.3 集团详情

集团详情按“一企一册”展示，建议分为以下页签：

| 页签 | 展示内容 |
|---|---|
| 基本信息 | 集团名称、编号、集团类型、BU、客户经理、重要集团、要客、139、本地/离岸、附属关系、统付信息、关键联系人 |
| 收入信息 | 当月收入、全年累计收入、分业务科目业务科目收入、收入趋势 |
| BC 业务信息 | 成员数、家宽数、专线数、物联网、移动云、IDC、ICT、5G 等业务指标 |
| 竞对信息 | 友商名称、业务类型、数量、统计月份、最近更新时间 |
| 商机信息 | 商机来源、名称、类型、状态、详情描述、创建时间 |
| 通讯录 | 联系人姓名、部门、职务、电话、状态、更新时间 |

处理规则：

- 收入信息按月度收入表汇总。
- BC 业务和竞对信息按最新统计月份展示，支持查看历史。
- 敏感联系方式根据权限进行脱敏展示。
- 集团关系信息通过集团关系表展示，不只依赖文本描述。

### 3.2.4 通讯录管理

| 项目 | 内容 |
|---|---|
| 功能说明 | 客户经理维护所辖集团的通讯录信息，支持新增、编辑、删除、批量导入，变更需经 BU 长审核后生效 |
| 使用角色 | 客户经理、BU 长 |
| 页面结构 | 两个 Tab：「生效中」展示已生效通讯录；「流程中」展示待审核 / 审核中 / 已驳回的通讯录 |
| 生效中列表 | 展示已生效的通讯录信息；操作列包含 [查看详情] [编辑] [删除] |
| 流程中列表 | 展示工单流程中的通讯录，状态 Tag 区分：待审核（蓝）/ 已驳回（橙）；操作列包含 [查看工单]；已驳回的记录客户经理可再次编辑后重新提交 |
| 新增 | 点击「新增联系人」按钮，弹出侧抽屉（Drawer），填写集团、姓名、部门、职务、联系电话后提交，生成通讯录变更工单 |
| 编辑 | 点击「编辑」按钮，弹出侧抽屉，修改字段后提交。客户经理编辑需 BU 长审核，若为已驳回记录则复用原工单重新提交；BU 长编辑本网格通讯录可直接生效 |
| 工单复用 | 已驳回的通讯录再次编辑提交时，系统复用原工单（工单号不变），追加快照记录变更内容，添加"重新提交"审批记录，工单状态从"驳回"变更为"待审核" |
| 删除 | 点击「删除」按钮，二次确认后生成删除工单，审核通过后逻辑删除 |
| 批量导入 | 点击「批量导入」按钮，弹出 Dialog，提供模板下载 → 文件上传 → 校验结果预览三步流程 |
| 查看详情 | 弹出侧抽屉，展示通讯录完整信息 + 历史变更记录（时间线） |
| 查看工单 | 弹出侧抽屉（640px），上半部分展示变更前后对比（修改：左右对比；新增：仅展示新数据；删除：仅展示待删除数据），下半部分展示工单流转时间线。若当前用户为 BU 长，底部展示审核意见表单 + [驳回] [通过] 按钮 |
| 处理规则 | 校验集团归属；校验手机号格式；同一集团内姓名+电话重复时提示；所有变更生成工单，审核通过后写入正式通讯录或执行逻辑删除；工单驳回时记录驳回原因 |
| 工单变更快照 | 提交时在 `t_yqyc_wf_ticket_contact_snapshot` 中记录变更前后内容，审核时展示快照不受正式表变化影响 |
| 权限控制 | 客户经理只能维护本人集团通讯录；BU 长可维护本网格集团通讯录，且可直接生效；BU 长可审核本网格工单 |
| 异常处理 | Excel 字段缺失、手机号格式错误、重复数据、非本人集团数据均进入失败明细；工单已被他人处理时提示状态已变化 |

| 项目 | 内容 |
|---|---|
| 功能说明 | 客户经理维护所辖集团的友商竞争信息，支持新增、编辑、删除，变更需经 BU 长审核后生效 |

| 异常处理 | 缺少业务科目、数量为负、统计月份格式错误时不允许提交；工单已被他人处理时提示状态已变化 |

## 3.3 BU 长端功能

### 3.3.1 BU 数据看板

| 项目 | 内容 |
|---|---|
| 功能说明 | 展示 BU 维度整体经营情况 |
| 使用角色 | BU 长 |

## 4. 系统技术架构

### 4.1 总体架构

```text
用户浏览器
  ↓
Vue 前端应用
  ↓ HTTP/HTTPS
后端 SSM 服务
  ├── Controller 接口层
  ├── Service 业务层
  ├── Workflow 流程服务
  ├── Auth 权限服务
  ├── Sync 数据同步服务
  └── Mapper 数据访问层
  ↓
数据库 MySQL / OceanBase
  ↓
外部系统
  ├── OA/统一认证
  ├── 营收数据源
  ├── 商机系统
  └── Excel 导入导出文件
```

### 4.2 前端架构

前端采用 Vue 技术栈，建议按以下结构设计：

| 模块 | 说明 |
|---|---|
| 路由模块 | 根据后端返回菜单动态生成路由，控制页面访问权限 |
| 状态管理 | 保存用户信息、角色、菜单、按钮权限、当前应用 |
| 看板组件 | 封装指标卡片、折线图、柱状图、排名榜 |
| 表格组件 | 支持分页、筛选、排序、导出入口 |
| 表单组件 | 支持集团、通讯录、竞对、字典等表单录入 |
| 导入组件 | 支持 Excel 上传、字段校验、失败明细展示 |
| 权限指令 | 控制按钮级权限，如新增、编辑、审核、导出 |

### 4.3 后端架构

后端采用 SSM，即 Spring + Spring MVC + MyBatis。

建议分层：

| 层级 | 职责 |
|---|---|
| Controller | 提供 REST 接口，处理参数校验和响应封装 |
| Service | 处理业务规则、权限校验、事务控制 |
| Mapper | 使用 MyBatis 访问数据库 |
| Domain/Entity | 数据库实体对象 |
| DTO/VO | 请求参数和返回对象 |
| Auth Service | 认证、菜单权限、数据权限过滤 |
| Workflow Service | 工单创建、状态流转、审批记录 |
| Sync Service | 收入、商机等外部数据同步 |
| Export Service | Excel 导入导出和异步导出任务 |

### 4.4 数据库架构

数据库测试阶段使用 MySQL，后续生产环境接入 OceanBase。设计原则：

1. 字段类型尽量使用 MySQL 与 OceanBase MySQL 模式通用类型。
2. 主键建议采用应用层统一生成的 BIGINT ID，降低数据库切换成本。
3. SQL 避免依赖数据库私有函数，复杂统计通过服务层或兼容 SQL 实现。
4. 所有核心查询建立必要索引。
5. 使用逻辑删除和审计字段，保留业务追溯能力。

### 4.5 安全架构

| 安全点 | 设计说明 |
|---|---|
| 身份认证 | 优先接入 OA/统一认证；本地登录只保存密码哈希和盐 |
| 访问控制 | 使用 RBAC 控制菜单、按钮、接口权限 |
| 数据权限 | 根据角色和组织过滤本人、本 BU、全部或自定义数据范围 |
| 敏感信息 | 手机号等联系方式按权限脱敏展示 |
| 操作审计 | 登录、导出、审核、权限变更、关键数据修改均记录日志 |
| 接口安全 | 参数校验、防越权、防重复提交、统一异常处理 |

### 4.6 集成架构

| 外部系统 | 集成内容 | 集成方式建议 |
|---|---|---|
| OA/统一认证 | 用户登录、账号基础信息 | 单点登录或接口同步 |
| 营收系统 | 月度收入、业务收入拆分 | 定时同步、批次入库 |
| 商机系统 | 商机来源、状态、描述 | 定时同步或接口查询 |
| Excel 文件 | 通讯录、集团清单、竞对信息导入 | 前端上传，后端校验入库 |

## 5. 优化后数据库设计

### 5.1 设计原则

1. 内部主键与业务编号分离：所有表使用 `id BIGINT` 作为内部主键，业务编号单独建唯一索引。
2. 字段规范统一：采用英文小写下划线字段名。
3. 关系清晰：通过 `group_id`、`user_id`、`org_id` 等内部 ID 建立关联。
4. 字典化：状态、类型、来源等枚举值通过字典表维护。
5. 支持审计：核心表包含 `created_by`、`created_time`、`updated_by`、`updated_time`、`deleted_flag`。
6. 支持扩展：预留应用、条线、数据权限和通用工单模型。

## 6. 核心业务流程

### 6.1 月度收入同步与看板展示流程

```text
营收数据源
  → 定时同步任务拉取收入数据
  → 按集团编号匹配 t_yqyc_crm_group
  → 按业务编码匹配 t_yqyc_biz_service
  → 写入 t_yqyc_biz_group_revenue_monthly
  → 记录同步批次和同步时间
  → 看板按用户数据权限汇总展示
```

处理要求：

1. 同步前校验集团编号和业务编码是否存在。
2. 同一集团、业务、月份、来源系统的数据重复同步时执行更新，不重复插入。
3. 同步失败记录错误明细，便于人工处理。
4. 看板查询必须叠加数据权限过滤。

### 6.2 通讯录导入与审核流程

```text
客户经理/BU 长上传 Excel 或单条录入
  → 系统校验字段、手机号、集团归属、重复数据
  → 生成通讯录变更快照
  → 创建 t_yqyc_wf_ticket 工单，状态为待审核
  → BU 长审核
      ├── 通过：写入/更新正式通讯录，工单完成
      └── 驳回：记录驳回原因，退回提交人
  → 全流程写入审批记录
```

处理要求：

1. Excel 必填字段包括集团编号或集团名称、姓名、联系电话。
2. 联系电话格式不正确的数据不得入库，应进入失败明细。
3. 审核通过后才进入正式通讯录。
4. 驳回原因必须填写。

### 6.3 友商竞对信息维护流程

```text
客户经理/BU 长录入竞对信息
  → 校验集团数据权限
  → 校验业务科目、友商、统计月份、数量
  → 保存竞对信息或创建变更工单
  → 记录操作日志
  → 集团详情和看板展示最新竞对信息
```

处理要求：

1. 数量不得为负数。
2. 同一集团、业务、友商、月份只能有一条有效数据。
3. BU 长可维护本 BU 数据，客户经理只能维护本人集团。
4. 所有修改需记录操作人和修改时间。

### 6.4 用户登录与数据权限流程

```text
用户登录
  → 校验账号状态
  → 加载用户角色
  → 加载菜单和按钮权限
  → 加载数据权限范围
  → 用户访问列表/详情接口
  → 后端按数据权限追加过滤条件
  → 返回授权范围内数据
```

处理要求：

1. 数据权限必须在后端执行，不能只依赖前端隐藏菜单。
2. 客户经理数据范围为本人集团。
3. BU 长数据范围为本 BU 集团。
4. 管理员数据范围为全量。

### 6.5 未来统一门户进入各平台流程

```text
员工登录统一门户
  → 门户查询用户可访问应用
  → 展示应用入口
  → 用户点击一企一册或其他平台
  → 携带统一认证票据进入目标应用
  → 目标应用加载本应用菜单和数据权限
```

处理要求：

1. 应用停用后不可进入。
2. 用户未授权应用不展示。
3. 不同平台可独立配置角色和菜单。
4. 统一门户保留应用访问日志。

## 7. 非功能需求

### 7.1 性能需求

| 场景 | 要求 |
|---|---|
| 普通列表查询 | 常规条件下 3 秒内返回 |
| 看板统计 | 常规条件下 5 秒内返回，可使用汇总表或缓存优化 |
| Excel 导入 | 支持批量导入，失败数据返回明细 |
| 数据导出 | 小数据量同步导出，大数据量异步导出 |
| 并发访问 | 支持部门内多客户经理和 BU 长同时使用 |

### 7.2 安全需求

1. 用户登录需校验账号状态。
2. 密码不得明文保存。
3. 手机号等敏感字段按权限脱敏。
4. 所有业务接口必须进行权限校验。
5. 数据导出必须记录日志。
6. 审核、权限变更、用户停用等关键操作必须审计。

### 7.3 可维护性需求

1. 业务状态、类型、来源等通过字典配置维护。
2. 菜单、按钮、角色、数据权限可配置。
3. MyBatis SQL 分模块维护，复杂统计 SQL 单独管理。
4. 前后端接口字段命名统一。
5. 日志和异常处理统一封装。

### 7.4 兼容性需求

1. 数据库兼容 MySQL 与 OceanBase MySQL 模式。
2. 前端兼容公司内常用浏览器环境。
3. Excel 导入模板应稳定，字段调整需版本化。
4. 外部系统同步失败不能影响用户登录和基础查询。

### 7.5 数据质量需求

1. 集团编号、业务编码、OA 账号、人力编号等关键字段需唯一。
2. 导入数据必须校验必填、格式、重复和权限。
3. 收入数据同步需有批次号和失败明细。
4. 历史数据迁移前需清洗集团编号、客户经理、BU 归属。

## 9. 风险与应对建议

| 风险 | 影响 | 应对建议 |
|---|---|---|
| 原始数据字段口径不统一 | 影响集团和收入统计准确性 | 建立字段字典和数据清洗规则 |
| 集团编号重复或变更 | 影响子表关联 | 使用内部 `group_id`，集团编号作为业务唯一键 |
| 收入数据同步失败 | 看板数据不准确 | 增加同步批次、失败明细和人工重跑机制 |
| 权限边界不清 | 可能造成越权查看 | 后端统一数据权限过滤，关键接口强校验 |
| OceanBase 兼容问题 | 后续数据库切换成本高 | 设计阶段避免数据库私有语法，提前进行兼容测试 |
| Excel 导入数据质量差 | 造成脏数据入库 | 导入前强校验，失败明细可下载 |
| 未来平台扩展导致重复建设 | 增加维护成本 | 当前即预留统一门户、应用、条线和通用工单模型 |

## 10. 总结

一企一册系统应以集团客户主数据为核心，以收入、业务、通讯录、竞对和商机为业务维度，以角色权限、数据权限和通用工单为支撑能力，形成客户经理、BU 长、管理员三类用户协同使用的信息化管理平台。

在数据库设计上，不建议直接沿用原需求中的草稿表结构，而应按本文提出的分域模型进行优化：

- 组织与权限域保障用户、角色、菜单和数据范围。
- 集团客户域保障集团主数据和通讯录。
- 业务经营域保障收入、BC、竞对和商机。
- 流程工单域保障通讯录和竞对等变更审核。
- 统一门户域承接未来公司级后台管理系统构想。
- 字典与日志域保障系统可维护、安全和可审计。

按该设计实施后，系统既能满足当前苏州移动战略客户中心的一企一册建设需求，也能逐步扩展为公司各条线共用的统一后台管理平台。
